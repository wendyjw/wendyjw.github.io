快速对比表

特性 Memory Cache (内存缓存) Disk Cache (磁盘缓存)
存储位置 RAM (内存) 硬盘 / SSD
访问速度 极快 (纳秒级) 快 (毫秒级，但比内存慢几个数量级)
生命周期 短暂 (临时性) 长久 (持久性)
关闭标签页后 缓存会消失 缓存仍然存在
容量 较小 较大
开发者控制 间接控制，主要由浏览器决定 直接控制 (通过 HTTP 缓存头)

详细解析

1. Memory Cache (内存缓存)

  • 是什么？ 它是将资源直接存储在计算机的 RAM (内存) 中的缓存。内存的读写速度远超硬盘，所以这是最快的缓存。

  • 生命周期 (关键特征)：
    它是临时性的，与当前页面的生命周期紧密相关。一旦你关闭了浏览器标签页或关闭了整个浏览器，这部分缓存就会被释放，数据会消失。

  • 它缓存什么？
    浏览器会倾向于将一些资源，特别是那些当前页面正在使用或刚刚使用过的资源放入内存缓存。这包括：

    • 预加载的资源 (preload)。
    • 已经解码的图片、解析后的脚本和样式表等，这些是“随时可用”的状态，无需再次处理。

  • 何时触发？

    • 当你在同一个标签页内进行页面跳转（比如从首页点到详情页，再点返回）。
    • 当一个资源在当前页面被多次请求时。

2. Disk Cache (磁盘缓存)

  • 是什么？ 它是将资源存储在你的硬盘或 SSD 上的缓存。它的速度虽然远不及内存，但对于网络请求来说，仍然是飞快的。

  • 生命周期 (关键特征)：
    它是持久性的。只要缓存没有过期、没有被用户手动清除、且没有超出容量限制，那么即使你关闭标签页或浏览器，甚至重启电脑，这部分缓存依然存在。

  • 它缓存什么？
    它会根据 HTTP 响应头中的规则，存储绝大多数的静态资源，如 JS、CSS、图片、字体等。这些资源是以原始文件的形式存储的。

  • 开发者如何控制？ (关键！)
    Disk Cache 是我们作为前端开发者可以直接控制的。通过在服务器上配置 HTTP 响应头，我们可以精确地告诉浏览器如何缓存文件：

    • Cache-Control: 这是最重要的头部，可以设置 max-age (缓存多久)、immutable (文件永不改变) 等。
    • ETag / Last-Modified: 用于协商缓存，当缓存过期后，浏览器可以用它们来询问服务器文件是否有更新。

两者如何协同工作？—— 缓存的优先级

浏览器在请求一个资源时，会按照以下优先级顺序进行查找：

  1. 第一步：检查 Memory Cache

    • 浏览器会首先检查内存中是否有这个资源的缓存。
    • 如果命中 ，则直接从内存中读取，速度最快。这是你在 DevTools 的 Network 面板中看到的 (memory cache)。

  2. 第二步：检查 Disk Cache

    • 如果内存缓存中未命中 ，浏览器会接着检查硬盘中是否有这个资源的缓存。
    • 如果命中，则从硬盘中读取。这个过程需要一次 I/O 操作，比内存慢，但仍然远快于网络请求。这是你在 Network 面板中看到的 (disk cache)。

  3. 第三步：发起网络请求

    • 如果内存和磁盘缓存都未命中，浏览器才会真正地向服务器发起网络请求。
    • 请求成功后，浏览器会根据该资源的 HTTP 响应头，决定是否以及如何将其存入 Disk Cache。同时，它可能会将这个刚下载的资源放入 Memory Cache，以备当前页面后续使用。

如何在 Chrome 开发者工具中观察？

  1. 打开开发者工具 (F12)，切换到 Network (网络) 标签页。
  2. **勾选 Disable cache**，然后访问你的网站，此时所有资源都是从网络加载的。
  3. **取消勾选 Disable cache，然后刷新页面 (F5)**。你会看到很多资源的 Size 列显示为 (disk cache)。
  4. 现在，在同一个标签页中，点击一个链接跳转到网站的另一个页面。然后再点击浏览器的“后退”按钮返回刚才的页面。再次观察 Network 面板，你可能会发现一些资源的 Size 列显示为 (memory cache)，因为浏览器认为这些资源马上会被再次用到，所以将它们保留在了更快的内存中。

总结： Memory Cache 是为了极致的速度和即时访问，服务于当前会话；而 Disk Cache 是为了长期的持久化和减少重复下载，服务于跨会话的用户体验。作为开发者，我们主要通过配置 HTTP 缓存头来优化 Disk Cache 的行为。
